## 6. Архітектура системи та залежності між підсистемами

Для забезпечення надійності, масштабованості та зручності супроводу обрано класичну трирівневу архітектуру (Three-Tier Architecture). Цей підхід передбачає чіткий логічний та фізичний розподіл системи на рівень представлення (Presentation Layer), рівень бізнес-логіки (Business Logic Layer) та рівень доступу до даних (Data Access Layer).

Рівень представлення реалізується у вигляді односторінкового веб-додатка (Single Page Application). Він відповідає за взаємодію з користувачем, рендеринг інтерфейсу, валідацію форм на боці клієнта та відображення динамічних даних без повного перезавантаження сторінки. Цей компонент взаємодіє з бекендом виключно через стандартизований REST API.

Рівень бізнес-логіки представлений сервером додатків. Тут зосереджені всі правила предметної області: обчислення допустимих навантажень, перевірка статусів, управління транзакціями та логіка авторизації. Цей рівень є повністю незалежним від інтерфейсу користувача, що дозволяє в майбутньому легко підключити до нього мобільні додатки або інтеграційні шлюзи партнерів. Крім того, на цьому рівні відбувається інтеграція із зовнішніми підсистемами, зокрема з API картографічних сервісів (Google Maps або Mapbox) для геокодування адрес та розрахунку транзитних відстаней.

Рівень даних складається з реляційної системи управління базами даних, яка забезпечує надійне довгострокове зберігання всієї інформації системи. Доступ до цього рівня має виключно сервер бізнес-логіки, що мінімізує ризики несанкціонованого втручання в дані.
